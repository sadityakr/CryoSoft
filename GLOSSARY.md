# CryoSoft Glossary

Canonical definitions of project terms. When a human and an agent (or two
agents) talk about CryoSoft, these are the meanings in force. If a term is
used in code, docs, or conversation with a different meaning than defined
here, the glossary wins; if a new recurring term appears, add it here in the
same commit that introduces it.

## Architecture layers (bottom to top)

| Term | Definition |
|---|---|
| **Layer (L0–L6)** | The layered architecture: L0 Driver → L1 Virtual Instrument → L2 Station/Config → L3 Orchestrator → L4 Procedure → L5 Data Manager → L6 Session Manager, with the GUI on top. Lower layers never import higher ones; the machine-checked rules live in `pyproject.toml` under `[tool.importlinter]`. |
| **Driver (real driver)** | L0. A Python class that talks to one physical instrument over PyVISA (GPIB/serial). Constructor takes exactly one required argument: the VISA resource string. Lives in `cryosoft/drivers/`, imports nothing from CryoSoft except `core.exceptions`. |
| **Sim driver** | A drop-in simulated twin of a real driver (`sim_` filename prefix), used for all testing without hardware. Must expose exactly the same public API as its real twin — enforced by `tests/test_conformance.py`. |
| **Virtual Instrument (VI)** | L1. A device-role abstraction (magnet, temperature controller, level meter, measurement) built on one or more drivers. VIs never import drivers; the Station constructs driver objects and injects them into the VI constructor as a `drivers` dict (dependency injection). All VIs subclass `BaseVirtualInstrument` and use `__init__(self, drivers, **init_params)`. |
| **Measurement method** | L1. A self-describing measurement VI (subclass of `MeasurementInstrumentBase`) that declares its own GUI knobs and output shape and implements one uniform lifecycle, so a generic procedure can run any of them without knowing the instrument or protocol. Self-description: `measurement_parameters` (name→`ParamSpec`, the single owner of the knobs), `measurement_data_keys` (the array names `take_reading()` returns), and optional `measurement_scalar_columns` (extra per-point scalars, e.g. `n_valid`). Lifecycle: `data_arrays(params)` (declared array lengths for the same params `initiate_measurement` receives), `initiate_measurement(**params)` (arm the hardware; `@control(panel=False)` where GUI-exposed; deliberately not named `initiate` — the plain lifecycle `initiate()` on a measurement VI is a harmless connection check, so Initiate-All can never start a source current), the argument-less `take_reading()` (one datapoint, always exactly the declared keys and lengths — a VI that can return fewer points pads with `float("nan")` and reports the true count in a scalar column), `standby()` (safe-off), and the optional **`reading_setters`** declaration (default `{}`; see **Reading loop**). `DCMeasurementBase` fixes the DC-resistance shape (`readings_per_point` samples of `voltage_V`/`current_A`); the full written standard lives in `MeasurementInstrumentBase`'s docstring and is machine-enforced by `test_conformance.py` (declaration validity, lifecycle presence, sim round-trip). |
| **Rotator VI** | L1. A VI over a motorized sample-rotation stage (class `vi_type = "rotator"`, registry `vi_type: system`), used with a uniaxial or 2D magnet to set the sample's orientation relative to the field. `RotatorVI` exposes exactly two controls/properties — `set_sample_angle(target_deg)` / `get_sample_angle()` and `set_rate_sample_angle(rate_deg_per_min)` / `get_rate_sample_angle()` — both bounded by the **control-validation standard** (`min_angle_deg`/`max_angle_deg`/`max_rate_deg_per_min` in config). Implements `RampableVI` with a status-driven ramp identical in shape to `SuperconductingMagnetVI`'s (waits for the driver to report `HOLD` before sending the next setpoint), so a rotator is a valid **system target** like a magnet or temperature controller. |
| **Switch VI** | L1. A VI over a matrix-switch / scanner driver (e.g. the Keithley 705) that multiplexes measurement channels by **route**. `SwitchMatrixVI` (class `vi_type = "switch"`, registry `vi_type: switch`) implements the **exclusive-mux policy**: at most one route is connected at a time, so `select_route(route)` always calls the driver's `open_all()` first and then closes the route's channels. `routes()` lists route names in config order; `@control select_route` / `@control open_all`; `get_state()` reports `active_route` (str, `""` when none) and `active_route_index` (int, `-1` when none). Route table and `settle_time_s` are validated at construction (`CryoSoftConfigError` naming the offender). A future true-matrix policy (multiple routes closed at once) would be a config flag, not a code fork. Participates in the **reading loop** via `reading_setters = {"route": "select_route"}` and `reading_safe_off = "open_all"`. |
| **Route** | A named connection on a **Switch VI**, its name taken verbatim from config, mapping to a list of instrument-format channel specs (e.g. `Mux-Ch1 -> ["1!1"]`). Route names must be non-empty, unique, and contain neither `__` (the column-label separator) nor `/` (illegal in an HDF5 dataset name). The route is the switch VI's loopable parameter (`reading_setters = {"route": "select_route"}`): put it in a **Reading loop** slot and tick the per-route pick checkboxes to measure several channels per sweep point, exactly like looping any other parameter. |
| **Reading loop** | The per-datapoint iteration a **generic sweep procedure** runs inside `measure()`: up to TWO generic **slots**, each a loopable parameter (see **`reading_setters`** — the switch VI's `route` and a measurement VI's `current_A` are the same concept) plus an ordered value list. Slot 1 (index labels `A1, A2, …`) is the outer level, slot 2 (`B1, B2, …`) the inner one. The auto-rendered "Reading loop" form group (shown above the measurement VI's parameter group whenever the station's reading path offers anything loopable) carries one structural `{slot}_parameter` drop-down per slot plus that parameter's values input — per-choice `{slot}_pick_{value}` checkboxes when the spec is enumerated (routes), a `{slot}_values` comma-separated text field otherwise (currents). A slot with ONE value is a static setting (dispatched once at `initiate()`, no suffix); with two or more it loops: the setter is dispatched as a `Command` through the Station before every reading and the columns compose as `{name}__A{i}__B{j}`. The label -> value map is stored in the run's HDF5 metadata (`procedure_params["loop_labels"]`). Non-measurement VIs that took part get their `reading_safe_off` (e.g. the switch's `open_all`) at standby/abort. A setup with no switch and no setters simply has no loopable parameters — no group, no special case. |
| **`reading_setters`** | The VI-level declaration behind the **reading loop**: a `BaseVirtualInstrument` class attribute mapping a parameter name to the cheap setter method that reprograms just that quantity between readings (e.g. the DC VI's `{"current_A": "set_source_current"}`, the switch VI's `{"route": "select_route"}`), paired with `reading_parameters` (the ParamSpec per loopable parameter — enumerated specs render as pick checkboxes, free specs as a value-list text field; a measurement VI inherits an implementation reading its `measurement_parameters`) and optional `reading_safe_off` (a method dispatched at standby/abort when a non-measurement VI took part, e.g. `open_all`). Declaring an entry is ALL a VI does — the "Reading loop" form group, the per-reading setter dispatch, the index-label column suffixes, and the metadata all follow automatically. Contract (machine-enforced by `test_conformance.py`): setters are real methods accepting the parameter under its own name and never change the reading's shape; measurement-VI keys are existing non-bool measurement parameters. |
| **Multiplexed columns** | The suffixed HDF5 columns a **generic sweep procedure** produces when a **reading loop** slot loops: every measurement array and per-point scalar is expanded per index label — slot 1 `A{i}`, slot 2 `B{j}`, composing as `{name}__A{i}__B{j}` (e.g. `voltage_V__A1__B2`) — via `DataSchema.multiplexed(labels, scalar_columns=...)` applied once per looping slot, slot 1 first. The metadata's `loop_labels` maps every label back to its value (route name, current, …). Non-loop columns (the sweep axis, system columns, `unix_time`) are never suffixed; with no looping slot the columns are plain. |
| **Station** | L2. The runtime registry of all VIs, built from a config directory by `build_station()`. The only layer that touches both VIs and (via config) driver classes. Dispatches system targets, measurement commands, and safety checks. |
| **Config** | A directory under `cryosoft/configs/<name>/` holding `devices.yaml` (drivers, VIs, wiring, safety limits) and `monitor.yaml` (tick interval, error threshold). Configs are the single source of truth for addresses, limits, and mappings — never hardcode these. |
| **Shipped vs user config** | *Shipped* configs are the git-tracked, read-only baselines under `cryosoft/configs/`; the GUI never edits them (a safe fallback that cannot be corrupted). *User* configs are editable copies under `%APPDATA%/CryoSoft/configs/`, created by copy-on-edit (`ConfigCatalog.fork_shipped`). |
| **Config catalog** | `core.config_catalog.ConfigCatalog` — discovers shipped + user configs, forks shipped→user, and keeps a named **config version** history per user config (save/list/restore). |
| **Config version** | A saved snapshot of a user config under `.versions/<id>/` with a `meta.json` (label + timestamp). Every editor save creates one; restoring copies it back over the active files. |
| **Active config** | The config the app loads on launch, stored in `QSettings` as an identity (`ActiveConfig/name` + `ActiveConfig/source`), re-resolved to a directory at startup so a stale absolute path (e.g. from a deleted git worktree) can't point nowhere. Switching it warns, saves the session, and restarts. |
| **Startup fallback chain** | `build_station_with_fallback()` tries the active config, then its shipped baseline, then the always-loadable `sim_cryostat`, so a corrupted config surfaces a warning instead of crashing startup. Each candidate is checked by `validate_config_dir()` (a dry-run that imports classes and resolves driver refs without instantiating hardware). |
| **Orchestrator** | L3. The state machine that executes a procedure: it ticks, ramps system VIs toward targets, waits, triggers measurements, and enforces safety. Talks only to the Station. Monitoring runs in its tick loop, gated by `start_monitoring()`/`stop_monitoring()` — see **Monitoring (Orchestrator)**. |
| **Procedure** | L4. A declarative measurement sequence (e.g. field sweep, temperature sweep) subclassing `BaseProcedure`. Declares *what* to do via the four-method interface; the Orchestrator decides *how*. Procedures never import drivers or VIs. Most measurement procedures are **generic sweep procedures** (see below), so adding a measurement usually means adding a measurement VI, not a procedure. |
| **Operation** | A multi-step cryostat-servicing action (subclass of `OperationBase`, L4): helium fill, sample change. Declarative like a procedure (returns plans, never touches VIs directly), but with operation-scope command access, tolerated safety flags, verified postconditions, an optional (not required) data file, and higher submission priority. Not listed among measurement procedures. |
| **Generic sweep procedure** | L4. `core.procedure.SweepMeasureProcedure` — the "sweep one axis, run any measurement method" base, and the two axis procedures on it: `FieldSweep` ("Field Sweep") and `TemperatureSweep` ("Temperature Sweep"). ONE procedure per sweep axis runs ANY station **measurement method**, chosen in the GUI via a **structural** `measurement_vi` parameter whose `choices` come from `Station.measurement_vi_names()`. The base owns measurement-VI selection, station-driven measurement-parameter merging, `DataSchema` assembly, per-datapoint validation, and the four-method loop; a concrete axis procedure supplies only the ramp targets, the axis read-back, and the settle times (its `_initial_system_targets` / `_step_targets` / `_standby_targets` / `_axis_readback` / `_initiate_wait_s` / `_step_wait_s` hooks). Replaces the former one-procedure-per-measurement classes (`FieldSweepDC`, `FieldSweepIV`, `TemperatureSweepDC`). |
| **Data Manager** | L5. Writes measurement data to HDF5 files. Standalone: knows nothing about instruments. Created by procedures in `initiate()`. |
| **Session Manager (L6)** | `cryosoft/session/` — the experiment-management layer between core and the GUI. `SessionManager` (the façade, `session/manager.py`) owns the **experiment** lifecycle, records every **run** automatically from the Orchestrator's **run manifests**, installs the experiment's **session envelope** on the Orchestrator, and supplies `experiment_context()` — a two-tier `{"setup": {...}, "experiment": {...}}` dict, see **Setup tier** — for stamping HDF5 files. Single writer of experiment state, exactly as the Orchestrator is the single writer to hardware. Imported by gui/main only; imports core downward (contracts C11/C12). |
| **GUI** | PyQt6 windows (monitor, procedure, instrument panels). Talks to the system only through the Orchestrator and Station APIs; never imports drivers or concrete VIs. |

## Runtime concepts

| Term | Definition |
|---|---|
| **Tick / tick loop** | The Orchestrator's periodic heartbeat (`tick_interval_ms` in `monitor.yaml`). Each tick processes queued GUI actions and advances the state machine/ramps; while **Monitoring (Orchestrator)** is active it also polls VI state and checks safety. |
| **Monitoring (Orchestrator)** | The hardware-touching part of the tick: polling every VI into `states_updated`, the safety watchdog, stale detection, and the operational-status record. OFF at construction so a fresh launch polls nothing until the instruments are initiated; started/stopped via `Orchestrator.start_monitoring()`/`stop_monitoring()` (the Monitor window's header toggle). `run_procedure()` auto-starts it; stopping is refused outside IDLE/ERROR. `shutdown()` stops the tick timer entirely (app exit / test teardown). |
| **System targets** | The `targets` field of a `PhasePlan`/`StepPlan`: a `{"vi_name": Target(...)}` mapping telling the Station which system VIs to ramp where. (Pre-Wave-2 this was a nested dict `{"vi_name": {"target": ..., "rate": ...}}`.) |
| **Measurement commands** | The `commands` field of a `PhasePlan`: an ordered `tuple[Command, ...]` the Station dispatches to measurement VIs. (Pre-Wave-2 this was a dict `{"vi_name": {"method": kwargs}}`.) |
| **Plan** | Umbrella term for the typed, frozen-dataclass currency in `core/plan.py` that procedures, the Orchestrator, and the Station exchange, replacing the old untyped dicts/tuples. Concretely: `Target`, `Command`, `PhasePlan`, `StepPlan` (plus the GUI/data-facing `ParamSpec`, `ParamGroup`, `DataSchema`). Each validates eagerly at construction, so a malformed plan fails at the boundary that built it. |
| **Target** | `core/plan.py`. A frozen dataclass naming a desired end-state for one system variable: `target` (SI value), optional `rate` (ramp rate forwarded to the VI's `start_ramp()` only when not `None`) and optional `persistent` (magnet persistent-mode flag). |
| **Command** | `core/plan.py`. A frozen dataclass naming one method call to dispatch on a VI: `vi_name`, `method`, `kwargs` (defensively copied). The typed request a procedure hands the Orchestrator, dispatched via `Station.send_measurement_commands()`. |
| **PhasePlan** | `core/plan.py`. What `initiate()` and `standby()` return: `targets` (name→`Target`), `commands` (ordered `tuple[Command, ...]`, never reordered), and `wait_s` (settle seconds). |
| **StepPlan** | `core/plan.py`. What `change_sweep_step()` returns for the next sweep point (`targets` + `wait_s`), or `None` when the sweep is done. |
| **ParamSpec** | `core/plan.py`. One GUI-facing procedure-parameter declaration (`type`, `default`, `unit`, bounds or enumerated `choices`, `structural`, `widget_hint`). Semantic only — names no Qt widget class. Procedures declare their `sweep_parameters` / `system_parameters` / `measurement_parameters` as `ParamSpec`s; the GUI maps each to a Qt widget in `gui/param_form.py`. |
| **Structural parameter** | A `ParamSpec` with `structural=True`: changing its value changes *which* parameter groups the form has (or which live-plot columns the run produces), so the GUI must re-derive the form. The generic sweep procedures declare them for `measurement_vi` (switching the measurement VI swaps the measurement parameter group) and every **reading loop** input (`loopN_parameter`, `loopN_values`, the `loopN_pick_*` checkboxes — changing them changes the loop levels and their live-plot label maps). `ProcedureWindow` re-renders only the changed group boxes (diffing `get_param_groups` output by `ParamGroup.key`), leaving the sweep/system boxes and the axis widget as the same widget instances. |
| **ParamGroup** | `core/plan.py`. One rendered sub-panel of a procedure's parameter form: a stable `key` (for value caching across re-renders), a `title`, and a name→`ParamSpec` `params` map. Returned by `BaseProcedure.get_param_groups(station, selections)` and rendered one `QGroupBox` per group by the GUI. A generic sweep procedure appends a `measurement_select` group (the `measurement_vi` selector) and a `measurement:{vi_name}` group (the selected VI's parameters) after its static Sweep/System groups. |
| **DataSchema** | `core/plan.py`. The declared HDF5 layout of one run (`sweep_columns` dtypes + `arrays` lengths); offers `multiplexed(labels, scalar_columns=())` (per-label array expansion — labels are the **reading loop**'s per-slot index labels, one call per looping slot — plus per-label expansion of the named scalar columns) and `validate(datapoint)` (reports every mismatch at once). A generic sweep procedure assembles one in `initiate()` (axis column + system columns + the selected VI's arrays and scalar columns) and validates every datapoint against it in `BaseProcedure._save_datapoint`; a mismatch raises `DataSchemaError`, which the Orchestrator contains to ERROR before anything is written. |
| **Ramp / RampableVI** | Gradual, rate-limited approach to a setpoint (field, temperature). VIs that support it implement the `RampableVI` interface (`start_ramp`, `advance_ramp`, `ramp_status`). |
| **`@monitored` / `@control`** | Decorators from `core/decorators.py` marking VI methods. `@monitored`: read-only, polled every tick into `get_state()`. `@control`: user-callable action, rendered as a button/input in the GUI instrument panel. |
| **Capability scope** | The tier a VI `@control` method belongs to: `"measurement"` (may appear in any plan) or `"operation"` (may appear only in operation plans; still a GUI control as before). Declared on the decorator, enforced at dispatch, conformance-checked. |
| **Control declaration** | The optional GUI metadata on `@control`: `params={name: ParamSpec}` (widget shape, unit, bounds, choices — validated against the signature at import) and `panel=` (default monitor-card placement; `False` = front-panel only). A VI whose valid values only exist after construction (e.g. a switch's config-named routes) overrides the instance hook `control_param_specs(method_name)` to inject dynamic choices; the GUI consults the hook, not the raw decorator metadata. Display-only — safety stays with `control_limits`. Conformance-checked. |
| **Panel control** | A `@control` shown on a VI's compact monitor card. The set is the VI's `panel=True` declarations, overridable per setup by the `panels:` block in `monitor.yaml`; every control, panel or not, appears in the instrument's front panel. |
| **Instrument front panel** | `gui/instrument_front_panel.py`. A per-VI child window (opened from the sliders icon on a card or device row) rendering the VI's complete surface: every `@monitored` value live and every `@control`, including panel-hidden ones. The GUI bench-test surface; all actions still flow through `Orchestrator.submit_vi_action()`. |
| **Shared-instrument mode discipline** | The standard (full text in `virtual_instruments/measurement/README.md`) for a driver whose public API spans more than one mutually exclusive operating mode (e.g. the Keithley 6221's plain DC output vs. its bipolar delta engine), when more than one measurement VI can be wired to the same physical driver instance in `devices.yaml`. The mode-establishing method MUST be idempotent and self-recovering — it unconditionally reasserts its own required mode (`Keithley6221.set_current()`'s `:SOUR:CURR:MODE FIX`) rather than assuming the instrument is already compatible, so a VI's `initiate()` never depends on a previous VI's `standby()` having run correctly. Teardown methods (`stop_delta_mode()`) still restore a documented idle baseline as defense-in-depth. A sim driver modeling more than one such mode tracks it (e.g. `SimKeithley6221._mode`) so skipping the defensive reassertion fails in tests instead of on hardware. |
| **Control-validation standard** | The binding contract (full text in `BaseVirtualInstrument`'s docstring) that every user-facing `@control` action is validated before hardware is touched and always yields an explicit verdict. (1) Numeric bounds: the VI declares `control_limits = {method: {param: limit_name}}` and its `__init__` populates `self._limits[limit_name] = (lo, hi)` from the config's `init_params` — limits are setup properties (e.g. `max_temperature_K`, `max_ramp_rate_K_per_min`, `max_current`), so a new cryostat changes YAML, not code. Enforcement is inherited: `__init_subclass__` wraps every `@control` method; an out-of-range call raises `CryoSoftSafetyError` with the reason. Subclasses adding limits must merge (`{**Parent.control_limits, ...}`). (2) Semantic rules that aren't ranges (e.g. never energise the switch heater across a PSU/coil current mismatch) are explicit checks at the top of the method, raising `CryoSoftSafetyError`. (3) Verdict: the Orchestrator emits `action_succeeded(vi, method)` or `action_failed(vi, method, reason)` for every submitted action; `MonitorWindow` shows failures in the notification banner and successes as a transient status-bar message. Machine-enforced by the `test_conformance.py` control-validation tests: declared limits must name real `@control` methods/params, must be populated by every sim-buildable config, and must actually reject an out-of-range call. |
| **vi_type (class)** | Class attribute of a VI describing its device role: `magnet`, `temperature`, `measurement`, `level`, `rotator`. Set by the typed base classes. |
| **vi_type (config/registry)** | The *registry* role a VI plays in a Station, set in `devices.yaml`: `system` (rampable, receives system targets), `measurement` (receives measurement commands), `level` (safety monitoring), or `switch` (a **Switch VI** that multiplexes measurement channels by route). Distinct from the class `vi_type`. |
| **`scanner_enabled`** | A `Station`-owned availability bit (`Station.set_scanner_enabled()`/`scanner_enabled()`) controlling whether the switch VI's loopable parameters are offered to the **reading loop**. Off by default. `Orchestrator.set_scanner_enabled()`/`scanner_enabled()` is the public toggle surface (a no-op, logged, when the station has no switch VI) — the bit lives on the Station because `SweepMeasureProcedure` only ever holds a `Station` reference, never an `Orchestrator` one. With the scanner disabled, a switch-VI-equipped station behaves to procedures exactly as if it had none: the route never appears in the loop-slot drop-downs. |
| **Offline instrument** | A configured VI whose hardware failed to *connect* when `build_station()` ran (driver `__init__` raised, or the VI's own bring-up raised a communication error). Recorded as an `OfflineInstrument` (name, registry `vi_type`, human-readable reason, failed driver aliases) in the Station's offline registry (`register_offline_vi()`, `offline_vi_names()`, `get_offline_info()`) instead of aborting the build — the degraded build. Invisible to the live enumerators (`get_vi_names()` etc.), so the Orchestrator, procedures, and safety evaluation see a smaller station; the GUI renders a control-free fault card (`gui/offline_panel.py`) whose detail window offers the one reconnect action, `Orchestrator.retry_reconnect()` → `Station.retry_instrument()` (IDLE-only, explicit verdict). Distinct from **disconnected** (`_disconnected`), which is a *runtime* poll-failure flag on a VI that did connect. Config errors are not offline instruments: they still abort the build and trigger the startup fallback chain. |
| **`Gate`** | `core/gates.py`. A tick-driven wait primitive: an optional one-shot `action` callable, followed by an optional `check` predicate that must hold True continuously for `window_s` seconds before `step()` reports satisfied (any `False` resets the stability clock). Polled once per Orchestrator tick, mirroring `RampableVI.advance_ramp()`/`ramp_status()`'s generator-per-tick idiom. |
| **`initiation_gates()` / `reading_gates()`** | `BaseProcedure` hooks (empty-tuple no-op default) returning `tuple[Gate, ...]`: gates that must all be satisfied before the Orchestrator proceeds from a completed ramp to a measurement. `initiation_gates()` governs the run's first RAMPING→MEASURING transition (right after `initiate()`); `reading_gates()` governs every one after (after each `change_sweep_step()`). When a procedure declares any gates for a transition, they fully replace that `PhasePlan`/`StepPlan`'s `wait_s` dwell timer for that transition — a procedure with no gates behaves identically to today's flat `wait_s` wait. |
| **`INITIATION_GATE` / `READING_GATE`** | `OrchestratorState` members entered from `RAMPING` once all system targets are reached and the active procedure declared gates for this transition; the Orchestrator steps every pending `Gate` once per tick until all are satisfied, then proceeds to `MEASURING`. Included in the same pausable/stale-checked state groups as `RAMPING`/`MEASURING`. |
| **Postcondition gate** | A `Gate` an operation declares via `postcondition_gates()`: stepped after parking completes, and the operation only reports **done** once every gate holds. The difference between "the commands ran" and "the cryostat is verifiably in the promised state". |
| **Persistent mode** | Superconducting-magnet operating mode where the field is held by a closed superconducting loop and the switch heater is off. Handled by `SuperconductingMagnetPersistentVI`. `start_ramp(target, persistent=True)`: `persistent=True` (default) ends the ramp by cooling the switch and parking the PSU at zero with the coil holding the field — for a single set-and-hold. `persistent=False` leaves the switch heater energised and the PSU holding the target directly — for a sequence of many ramps (e.g. a field sweep) that should only pay the heater warmup once. `get_field()`/`magnet_current()` read the coil current instead of the PSU current whenever the driver reports persistent mode. |
| **VISA / GPIB** | The instrument-communication standard (PyVISA library) and the bus most lab instruments use. A "resource string" like `GPIB0::19::INSTR` addresses one instrument. |
| **SI units rule** | All APIs use Tesla, Kelvin, Ampere, Volt, second. Display formatting (mK, µA) happens only in the GUI. |
| **Sweep builder** | `core/sweep_builder.py` — reusable sweep-array construction shared by Procedures: `build_piecewise_sweep()` (a `SweepSegment(start, end, step)` list stitched into one array, letting different sub-ranges use different step sizes, e.g. coarse outside a region and fine inside it), `load_custom_sweep_csv()` (arbitrary non-uniform sweep from a single-column CSV), and `apply_hysteresis()` (extends a one-directional sweep into a forward+backward loop). See **Sweep axis** for the declarative way most Procedures use these three. |
| **Experiment** | A named group of **runs** on one sample toward one scientific question, owned by the Session Manager (L6): `ExperimentRecord` holds the **user**, the sample-info snapshot, the config identity, the **session envelope**, the run list, free-text findings, the GUI-authored run `queue` (opaque JSON the session layer stores but never interprets — contract C11), and (publishing track) the ELN linkage. Persisted as `<sessions_root>/<experiment_id>/experiment.json` (`sessions_root` = `cryosoft.gui.app_settings.sessions_root()`, an app-settings-backed location independent of any GUI form field, default `<Documents>/CryoData`), alongside a `gui_state.json` (GUI form/plot state, also opaque to this layer) and a `data/` folder holding the experiment's HDF5 files (sub-folders allowed, e.g. `data/heating_runs/`) — the whole `<experiment_id>/` folder is the complete, portable record. `RunRecord.data_file` is stored relative to that folder when the file lives inside it (`ExperimentStore.relativize_data_file()`/`resolve_data_file()`), so the folder can be copied or moved and still resolve; a file saved deliberately outside it stays an absolute path. Every persisted file (`experiment.json`, `gui_state.json`, `active.json`) carries a top-level `schema_version` (`models.SCHEMA_VERSION`); a value newer than the running app's is loaded tolerantly but treated read-only (never re-saved). The `active.json` pointer resumes the open experiment across restarts; `SessionManager.switch_experiment()` moves between several open experiments without closing the one switched away from. One experiment ↔ one ELN entry. The Experiment tier — as opposed to the **Setup tier** — is what `SessionManager.experiment_context()`'s `"experiment"` key holds; empty (`{}`) when none is open. |
| **Setup tier** | The metadata that describes the rig, not any one experiment: which config is active, each VI's **instrument metadata**, and who's logged in. Changes rarely (switching config/instrument description/user), unlike the Experiment tier which changes per run. Not owned by any one L6 class — it's assembled from `core.station.read_instrument_metadata()` + the config's identity (both cached by `SessionManager` at construction) and `app_settings.current_user_id()`/`UserRoster` (GUI-level). Surfaced in the GUI through menus (User, Config), not a docked panel — see **User menu**. `SessionManager.experiment_context()`'s `"setup"` key holds it, present even with no experiment open. |
| **Instrument metadata** | An optional `metadata:` block per VI in `devices.yaml` (free-text fields — `role`, `manufacturer`, `model`, `notes`, whatever the setup wants to record about what a VI physically is). A setup property like everything else in `devices.yaml`; edited via the Config Editor, never in the running app. Read by `core.station.read_instrument_metadata(config_path)` — YAML-parse only, never imports a driver/VI class, so it's safe to call from the GUI thread. Displayed read-only in `InstrumentInfoDialog` and stamped into every run's `experiment_info["setup"]["instruments"]`. |
| **Run** | One execution of one procedure, producing exactly one HDF5 file. Typed as `RunRecord` (L6): id, procedure, kind (`run`/`probe`), params, data-file path, timestamps, terminal status (`done`/`failed`/`aborted`), and the initiate-time instrument-settings snapshot. Recorded automatically from the Orchestrator's **run manifests**; runs outside an open experiment are not recorded (their HDF5 file still exists, unstamped). |
| **User** | The person measuring — identity, not authentication. `User` records (id, name, email, ORCID, ELN identity) live in the setup-local roster `users.json` next to the app settings, managed by `UserRoster`. Experiments reference a roster `user_id`, and the id/name are stamped into every run's HDF5 `experiment_info`. |
| **Attendance** | The per-experiment attended/unattended flag (`ExperimentRecord.attended`, set via `SessionManager.set_attended()`). Governs how much autonomy debug agents get in the agent-native track: gentle recovery actions are reserved for unattended sessions; attended sessions get diagnose-and-report only. |
| **Run manifest** | The dict the Orchestrator emits on `run_started` (once per run, after a successful `initiate()`) and re-emits on `run_finished` (with `finished_utc`, terminal `status` done/aborted/failed, and `reason` added). Carries `run_id`, `procedure`, `kind` (`run`; probe runs will carry `probe`), `params`, `data_file` (the HDF5 path, captured at start because the procedure closes its file before the run ends), and `started_utc`. The session layer records runs from these signals; a run whose setup fails emits no manifest. |
| **Session envelope** | `core/plan.py`'s `SessionEnvelope` (+ `EnvelopeBound`): per-experiment safety bounds narrower than the config limits, keyed by system VI name. Config `init_params` limits protect the *instrument* (setup property, YAML); the envelope protects the *sample* mounted for one experiment (experiment property, set by the session layer via `Orchestrator.set_session_envelope()`). Enforced in the Orchestrator so it binds every writer — targets are validated before dispatch (violation ⇒ the plan is rejected), and each bound with a `state_key` is checked against the VI's live reading every tick (violation ⇒ EMERGENCY, exactly like a tripped safety flag). |
| **Servicing log** | The L6 framework for per-setup, human-facing logs of servicing events. Each log is one **log kind**; entries are typed records, editable via **entry revisions**, persisted append-only. Independent of experiments; what technical staff consult and maintain. |
| **Log kind** | One declared servicing-log table: a key (`"cryogenics"`), a title, and an ordered field schema (reusing `ParamSpec`). Adding a log kind for a new setup is a declaration + config reference — the storage, revision handling, table view, and add/edit dialogs all follow automatically. The standards-over-one-off-code move, applied to record keeping. |
| **Cryogenics log** | The first log kind: one entry per cryogen fill — `person`, `start_utc`, `end_utc`, `helium_start_pct`, `helium_end_pct`, `ln2_filled` (bool), `notes`. Written automatically by the fill operation, addable and editable manually (fills done by hand, LN2 top-ups, corrections). |
| **Entry revision** | How editability and trustworthiness coexist: every entry has a stable `entry_id`; an edit appends a new revision (`revised_utc`, `revised_by`) rather than rewriting; a deletion appends a tombstone. The store presents the latest revision per entry; the full history stays on disk. The file itself is never rewritten. |
| **Helium record** | The machine companion to the cryogenics log: one `(utc, helium_pct, nitrogen_pct)` sample per hour, appended by the recorder from the monitoring tick. Powers the consumption display and start/end-level lookup; not human-editable. |
| **Helium fill (operation)** | `procedures/operations/helium_fill.py`'s `HeliumFillOperation` — the first concrete `OperationBase` (plan §8.1): ramps every magnet (`Station.magnet_vi_names()`) to zero field (`initiation_gates()` holds until zero field is confirmed and held), switches the level meter to FAST refresh, samples the helium level once per `sample_period_s` into an HDF5 curve, and finishes once the level has held at/above `fill_target_pct` and non-rising for `fill_complete_window_s` (or `max_fill_duration_s` elapses). `standby()`/`abort()` restore SLOW refresh so an aborted fill never leaves the meter in FAST; `postcondition_gates()` verifies that restoration and that the level did not fall below its start value. Tolerates `helium_low` (`tolerated_safety_flags`) — the fill's whole purpose. Configured from the `cryogenics:` block (`core.station.read_cryogenics_config()`); on finish, `CryogenicsRecorder` composes the **cryogenics log** entry from the manifest and the fill's own sampled start/end levels. |
| **Sample change (operation)** | `procedures/operations/sample_change.py`'s `SampleChangeOperation` — the second concrete `OperationBase` (plan §8.2): "verify the cryostat is safe to open". `initiate()` ramps every magnet (`Station.magnet_vi_names()`) to zero field and the configured VTI VI to `target_temperature_K` (default 300 K), sends `open_all` to the first switch VI (if any) and `standby` to every measurement VI (`Station.measurement_vi_names()`); no `initiation_gates()` and no sampling loop (`step()` returns `None` immediately) — the run's whole duration is carried by the ramps and by `postcondition_gates()`. `postcondition_gates()` reads only cached state: `zero_field` (every magnet, held `zero_field_window_s`); `heater_off` (only for magnets whose cached state exposes `switch_heater_state` — a plain `SuperconductingMagnetVI` has none and is skipped; omitted entirely if no magnet exposes the field); `vti_at_target` (held `temperature_window_s`); and, when `needle_valve == "manual"` (the only supported mode today — a VI-capability reference is future work), `needle_valve_confirmed`, gated on an **operator confirmation**. `tolerated_safety_flags` is left empty — a sample change has no business running under an active safety condition. Configured from the `operations.sample_change:` block (`core.station.read_operations_config()`). |
| **Operator confirmation** | A human attestation an operation's `postcondition_gates()` can gate on when no machine-verifiable capability exists (e.g. `SampleChangeOperation`'s manual needle valve, plan §8.2). An operation declares a class-level `operator_confirmations: dict[str, str]` (key → checkbox label); `confirm(key)` / `confirmed(key)` set and read the flag; `Orchestrator.confirm_operation(key)` (mirroring `finish_operation()`) is the GUI's entry point, duck-typed exactly like `finish_operation()`'s `request_finish()`. Recorded in the **operations stream** as human-confirmed rather than machine-verified. |
| **Readiness condition** | `core/operation.py`'s `ReadinessCondition` (a frozen dataclass: `key`, `label`, `check(state) -> bool`, optional `detail(state) -> str`) — one live checklist row the Operations panel renders for an operation, declared via `OperationBase.readiness_conditions() -> tuple[ReadinessCondition, ...]` (plan §12, default `()`). `check`/`detail` are pure reads against the Orchestrator's per-tick state snapshot (`{vi_name: {field: value}}`) — no hardware access, no extra poll. E.g. `HeliumFillOperation`'s `zero_field` row; `SampleChangeOperation`'s `zero_field`/`heater_off`/`vti_at_target`/`needle_valve_confirmed` rows. |
| **Next due** | `core/operation.py`'s `NextDue` (a frozen dataclass: `due_unix: float \| None`, `text: str`) — an operation's schedule prediction, returned by `OperationBase.next_due(context) -> NextDue \| None` (plan §12, default `None` — no schedule). `context` is a GUI-assembled dict (`"state"`, `"now_unix"`, `"consumption_rate_pct_per_h"`); an operation must not import the session layer to compute its own rate (contract C12), which is why the rate is passed in rather than computed in `next_due()`. `HeliumFillOperation.next_due()` predicts time-to-`helium_warning_pct` from the measured consumption rate; `SampleChangeOperation` has none. |
| **config_key** | `OperationBase`'s class attribute (`str`, default `""`, plan §12) naming the `operations:` config-block key this operation class should be built from generically — e.g. `SampleChangeOperation.config_key = "sample_change"` matches an `operations.sample_change:` block. The Operations panel maps every declared `operations:` key to the discovered class whose `config_key` matches (unknown keys log a warning and are skipped); a non-empty `config_key` must be unique across operations (checked by `tests/test_conformance.py`). The helium fill leaves it `""` — it is wired from `cryogenics:`, not the generic `operations:` lookup. |
| **Sweep axis** | `core/sweep_builder.py`'s `SweepAxis` dataclass + `sweep_axis_param_specs()` + `build_axis_sweep()` — the systematic, low-code way a Procedure gets sweep-shape flexibility (linear range / piecewise segments for a fine subfield / custom CSV, each optionally with hysteresis). A Procedure declares one `sweep_axis = SweepAxis(key=..., unit=..., data_key=..., description=..., default_start=..., default_end=..., default_steps=...)` class attribute (see `FieldSweep`, `TemperatureSweep`); `BaseProcedure.__init_subclass__` merges its seven hidden parameters (`{key}_mode`, `{key}_start`, `{key}_end`, `{key}_steps`, `{key}_segments`, `{key}_csv_path`, `{key}_hysteresis`) into `cls.parameters`, and the default `_build_sweep_array()` delegates to `build_axis_sweep()` — no per-procedure sweep-array code is needed. `gui/sweep_axis_widget.py`'s `SweepAxisWidget` renders the matching mode-selector form (Linear / Segments / CSV + a hysteresis checkbox) automatically in `ProcedureWindow`; a new sweep procedure never needs its own GUI code, only the `sweep_axis` declaration. |

## GUI concepts

| Term | Definition |
|---|---|
| **Quadrant layout** | The fixed 2x2 grid both `MonitorWindow` and `ProcedureWindow` use below their header/banner: a `main_splitter` (`QSplitter(Horizontal)`) containing `left_splitter` and `right_splitter` (each `QSplitter(Vertical)`), giving four panes whose boundaries are draggable but which can never be closed, detached, or floated — unlike the `QDockWidget` layout it replaced, which let panels overlap, hide each other, and become glitchy to resize. In `MonitorWindow`: top-left Instruments (scrollable 2-column `InstrumentPanel` grid holding EVERY VI — system/level plus role-tagged measurement and switch cards), top-right Trends, bottom-left the `SessionInfoPanel` (Experiment tier: status/Start-Close/attendance/findings, sample fields, eLab status), bottom-right the optional `OperationsPanel`. In `ProcedureWindow`: top-left the parameter form (filling the quadrant, not capped at a fixed height), top-right the Queue, bottom-left/right Plot 1/Plot 2. Splitter proportions persist via `QSettings` alongside window geometry, restored defensively — see `MonitorWindow._restore_splitter_state()`'s shape-mismatch guard, which discards a restored blob whose orientation/child-count doesn't match (protects against a stale value under a reused settings key, e.g. from a since-removed earlier layout). |
| **Operations panel** | `gui/operations_panel.py`'s `OperationsPanel` (plan §12) — `MonitorWindow`'s bottom-right "Operations" selector entry (renamed from "Cryogenics", which it absorbs, not replaces), available when cryogenics is enabled OR the `operations:` config is non-empty. An optional cryogenics status section (levels/consumption/plot, gated on `cryogenics:`) followed by one generic `OperationCard` per available operation — the helium fill (iff cryogenics is configured) plus one per `operations:` config block whose `config_key` matches a class from `gui/procedure_discovery.py`'s `discover_operations()`. Answers, at a glance: what state is the setup in, is it ready right now for operation X, and when will X next be needed? |
| **OperationCard** | `gui/operations_panel.py`'s `OperationCard` — one operation's fully generic panel widget (plan §12): a header (name + **next due** line, hidden when `next_due()` returns `None`), one row per **readiness condition** (colored ✓/✗ icon, label, live detail text, re-evaluated every `on_states_updated` tick), an operator-confirmations row (hidden unless this operation is the active run and declares `operator_confirmations`), a **ready banner**, and a start/finish button. Built from one *display* operation instance (constructed once, for readiness/next-due/labels only, never run) plus a panel-supplied *factory* closure that builds a FRESH instance per actual run — this is what keeps the class free of any per-operation code. |
| **Ready banner** | `OperationCard`'s prominent success message (plan §12): shown once a run of that card's operation has finished `done` AND every current **readiness condition** holds, using the operation's `ready_message` class attribute (e.g. `"Ready — sample can be taken out"`; empty = no banner, ever). Clears the instant a condition stops holding or a new run starts (`request_finish`/`run_started`) — it never shows a stale "ready" after the setup has moved on. |
| **User menu** | `MonitorWindow`'s leftmost menu (`monitor_window.py`), holding **Setup tier** concerns: `Log in as…` (opens `setup_dialogs.LoginDialog`, identity only — no password), `Load Session…` (opens `session_dialogs.LoadSessionDialog` to switch the open L6 experiment via `SessionManager.switch_experiment`/`MonitorWindow._switch_session`), `Sessions Folder…` (browses `app_settings.sessions_root()`/`set_sessions_root()` — applies fully on next launch; the running `ExperimentStore` keeps its old root), and `New Session`/`Save Session Now` (operate on `form_autosave`). Named "User", not "Session" — a "Session" menu existed here once, but the word was deliberately freed for the L6 experiment layer (`form_autosave.py` was `session.py`); reusing it for this menu would recreate exactly that ambiguity. |
| **Current user (login)** | `app_settings.current_user_id()`/`set_current_user_id()` — identity only, no password, persisted in `QSettings` like the active config. Governs which per-user `form_autosave` file (`app_settings.session_file_path(user_id)`, under `sessions/<user_id>.json`) `SessionInfoPanel`'s sample fields load from/save to, so switching users switches what's remembered instead of one person's fields overwriting another's. `MonitorWindow._switch_user()` does the save-outgoing/load-incoming swap; the header's `current_user_label` reflects it. Independent of any experiment's `user_id` (`ExperimentRecord.user_id`, assigned at Start Experiment) — logging in does not open or require an experiment. |
| **Form autosave** | `cryosoft/gui/form_autosave.py` — the GUI's persisted form content (sample metadata, data dir, last procedure + params, run queue), written to one JSON file so a reopened app restores what the physicist typed. Historically named the "session model" (`gui/session.py`); renamed so **session** unambiguously means the L6 Session Management concept. The classes (`SessionState`) and the `last_session.json` file keep their old names for compatibility. |
| **MonitorHistory** | `cryosoft/gui/monitor_history.py`. A Qt-free, pure-Python ring buffer that accumulates flattened `{vi_name}_{field_name}` time-series history from `Orchestrator.states_updated`, one bounded `deque` per key, for the Trends panels' plots. |
| **Trends grid** | 1–4 `TrendPlotPanel`s arranged in the Trends quadrant's `QGridLayout`, auto-recomputed to `ceil(sqrt(N))` columns on every add/remove (1→1x1, 2→2x1, 3–4→2x2). The quadrant's "Add trend plot" button adds panels up to a cap of 4 (disabling once reached); each panel's own remove button drops it, never below a floor of 1. The two default panels opportunistically select a temperature/level reading (not a setting/rate field) once `MonitorHistory` has data. Each panel's Y-variable `QComboBox` uses `SizeAdjustPolicy.AdjustToContents` and a layout stretch factor of 1 so long flat keys (e.g. `temperature_sample_temperature`) get first claim on the row's width, ahead of the time-window selector and remove button. |
| **LifecycleToggleButton** | `cryosoft/gui/lifecycle_toggle.py`. A single state-dependent button + status glow dot replacing the old separate Initiate/Standby `QPushButton` pair, shared by `InstrumentPanel` (in its custom header row next to the VI name, since `QGroupBox` can't embed a widget next to its native title), present on every card. Clicking only submits the opposite action (`"initiate"`/`"standby"`) via a callback — it never flips state optimistically. `set_initiated(bool)` is the only thing that changes the displayed state (text, icon, dot colour), and both call sites wire it to `Orchestrator.action_succeeded`, so the glow reflects confirmed instrument state, not a hopeful click. |

## Development harness

| Term | Definition |
|---|---|
| **Layer contracts (C1–C12)** | The machine-checked import rules in `pyproject.toml` `[tool.importlinter]`. Run with `make contracts`. A broken contract means the change crosses a layer boundary — route through the proper interface instead of editing the contract. C9/C10 isolate the troubleshoot toolbox in both directions; C11/C12 place the session layer (L6) between core and the GUI. |
| **Troubleshoot toolbox** | `cryosoft/troubleshoot/` — the diagnostic package (engine + CLI) for commissioning and debugging a setup. A leaf entry point beside the stack, like `main.py`: it may import drivers and the Station's config helpers; nothing in cryosoft imports it. Run while the main app is closed (serial ports are exclusive-open). CLI: `python -m cryosoft.troubleshoot scan/probe/check/methods/idn/read/write/query/send`, each one-shot with `--json` and exit code 0/1; `write`/`query`/`send` are the permission-gated verbs. Every invocation appends to the transcript `cryosoft/logs/troubleshoot.jsonl`. |
| **Fault taxonomy / FaultCode** | The stable machine-readable outcome codes every troubleshoot probe returns (`OK`, `CONFIG_INVALID`, `ADDRESS_NOT_ON_BUS`, `OPEN_FAILED`, `NO_RESPONSE`, `WRONG_IDN`, `GARBLED_RESPONSE`, `DRIVER_ERROR`). Codes are API for the triage skill: never renamed, only extended. |
| **Config preflight** | `check_config()` in the troubleshoot engine: for every `real_drivers` entry, verify bus presence, construct the driver, call `get_idn()`, and optionally match the config's `expect_idn` substring. The "is everything connected?" check run before a cooldown. |
| **Driver bench** | `DriverBench` in the troubleshoot engine: wraps one driver instance for method introspection, read/write-gated calls, and raw SCPI query/send. The primitive behind driver development and per-instrument debugging. |
| **`expect_idn`** | Optional key on a `real_drivers` entry in `devices.yaml`: a case-insensitive substring the instrument's `get_idn()` reply must contain. Lets the preflight detect swapped addresses/cables (`WRONG_IDN`). |
| **`get_idn()` (driver contract)** | Every driver — sim and real — exposes `get_idn()` taking no arguments, returning an identification string. Pre-SCPI Oxford instruments implement it with their native `V` command. Enforced by conformance. |
| **setup.md** | Per-config ground-truth document (`<config dir>/setup.md`): instrument purposes, wiring notes, dated known quirks, per-setup safe-testing limits. Written once during commissioning (human-confirmed), maintained by agents after every diagnosis. Template in `.claude/skills/setup-commission/references/`. |
| **Safe-testing ladder** | The excitation discipline for diagnostics (`.claude/skills/setup-supervisor/references/safe-testing.md`): L0 passive → L1 zero-excitation set/read-back → L2 minimal excitation (≤10 mA magnet lead current, ≤1 nA precision sources) → L3 operational values (human-confirmed state only). Instruments are not code: damage is irreversible. |
| **Incident report** | `cryosoft/logs/incidents/YYYY-MM-DD-<slug>.md` — the structured "cannot conclude" exit of the setup-supervisor skill (evidence table, layers ruled out, hypothesis with confidence, recommended next steps). A good report is a successful outcome; a guess is not. |
| **Conformance tests** | `tests/test_conformance.py`. Auto-discover every driver, VI, procedure, and config and check they follow their interface contracts. A new module is covered automatically, with no test-writing needed. |
| **`make check`** | The blocking quality gate (lint + contracts + tests). Run it before declaring any work done; CI runs exactly the same targets. |
| **`hardware` marker** | Pytest marker for tests needing physical instruments. Excluded by `make test` and CI; run manually at the cryostat. |
| **LOGBOOK.md** | The running development log at the project root (newest first, not git-tracked). Every work session — human or agent — ends by prepending an entry. |
