# CryoSoft Glossary

Canonical definitions of project terms. When a human and an agent (or two
agents) talk about CryoSoft, these are the meanings in force. If a term is
used in code, docs, or conversation with a different meaning than defined
here, the glossary wins; if a new recurring term appears, add it here in the
same commit that introduces it.

## Architecture layers (bottom to top)

| Term | Definition |
|---|---|
| **Layer (L0–L6)** | The layered architecture: L0 Driver → L1 Virtual Instrument → L2 Station/Config → L3 Orchestrator → L4 Procedure → L5 Data Manager → L6 Session Manager, with the GUI on top. Lower layers never import higher ones; the machine-checked rules live in `pyproject.toml` under `[tool.importlinter]`. |
| **Driver (real driver)** | L0. A Python class that talks to one physical instrument over PyVISA (GPIB/serial). Constructor takes exactly one required argument: the VISA resource string. Lives in `cryosoft/drivers/`, imports nothing from CryoSoft except `core.exceptions`. |
| **Sim driver** | A drop-in simulated twin of a real driver (`sim_` filename prefix), used for all testing without hardware. Must expose exactly the same public API as its real twin — enforced by `tests/test_conformance.py`. |
| **Virtual Instrument (VI)** | L1. A device-role abstraction (magnet, temperature controller, level meter, measurement) built on one or more drivers. VIs never import drivers; the Station constructs driver objects and injects them into the VI constructor as a `drivers` dict (dependency injection). All VIs subclass `BaseVirtualInstrument` and use `__init__(self, drivers, **init_params)`. |
| **Measurement method** | L1. A self-describing measurement VI (subclass of `MeasurementInstrumentBase`) that declares its own GUI knobs and output shape and implements one uniform lifecycle, so a generic procedure can run any of them without knowing the instrument or protocol. Self-description: `measurement_parameters` (name→`ParamSpec`, the single owner of the knobs), `measurement_data_keys` (the array names `take_reading()` returns), and optional `measurement_scalar_columns` (extra per-point scalars, e.g. `n_valid`). Lifecycle: `data_arrays(params)` (declared array lengths for the same params `initiate` receives), `initiate(**params)` (arm the hardware; `@control` where GUI-exposed), the argument-less `take_reading()` (one datapoint, always exactly the declared keys and lengths — a VI that can return fewer points pads with `float("nan")` and reports the true count in a scalar column), and `standby()` (safe-off). `DCMeasurementBase` fixes the DC-resistance shape (`readings_per_point` samples of `voltage_V`/`current_A`); the full written standard lives in `MeasurementInstrumentBase`'s docstring and is machine-enforced by `test_conformance.py` (declaration validity, lifecycle presence, sim round-trip). |
| **Switch VI** | L1. A VI over a matrix-switch / scanner driver (e.g. the Keithley 705) that multiplexes measurement channels by **route**. `SwitchMatrixVI` (class `vi_type = "switch"`, registry `vi_type: switch`) implements the **exclusive-mux policy**: at most one route is connected at a time, so `select_route(route)` always calls the driver's `open_all()` first and then closes the route's channels. `routes()` lists route names in config order; `@control select_route` / `@control open_all`; `get_state()` reports `active_route` (str, `""` when none) and `active_route_index` (int, `-1` when none). Route table and `settle_time_s` are validated at construction (`CryoSoftConfigError` naming the offender). A future true-matrix policy (multiple routes closed at once) would be a config flag, not a code fork. |
| **Route** | A named connection on a **Switch VI**, its name taken verbatim from config, mapping to a list of instrument-format channel specs (e.g. `Mux-Ch1 -> ["1!1"]`). Route names must be non-empty, unique, and contain neither `__` (the array/route separator) nor `/` (illegal in an HDF5 dataset name). A **generic sweep procedure** renders one "Measure route <name>" checkbox per route (form param `mux_<route>`), connects the first selected route before arming the measurement VI at `initiate()`, and — with two or more routes selected — re-selects each route per datapoint in `measure()`. |
| **Multiplexed columns** | The per-route HDF5 columns a **generic sweep procedure** produces when two or more **routes** are selected: every measurement array and per-point scalar is suffixed `{name}__{route}` (e.g. `voltage_V__Mux-Ch1`, `n_valid__Mux-Ch2`) via `DataSchema.multiplexed(routes, scalar_columns=...)`. Non-mux columns (the sweep axis, system columns, `unix_time`) are never suffixed. With 0 or 1 route the columns are unsuffixed (exactly as before switch support). |
| **Station** | L2. The runtime registry of all VIs, built from a config directory by `build_station()`. The only layer that touches both VIs and (via config) driver classes. Dispatches system targets, measurement commands, and safety checks. |
| **Config** | A directory under `cryosoft/configs/<name>/` holding `devices.yaml` (drivers, VIs, wiring, safety limits) and `monitor.yaml` (tick interval, error threshold). Configs are the single source of truth for addresses, limits, and mappings — never hardcode these. |
| **Shipped vs user config** | *Shipped* configs are the git-tracked, read-only baselines under `cryosoft/configs/`; the GUI never edits them (a safe fallback that cannot be corrupted). *User* configs are editable copies under `%APPDATA%/CryoSoft/configs/`, created by copy-on-edit (`ConfigCatalog.fork_shipped`). |
| **Config catalog** | `core.config_catalog.ConfigCatalog` — discovers shipped + user configs, forks shipped→user, and keeps a named **config version** history per user config (save/list/restore). |
| **Config version** | A saved snapshot of a user config under `.versions/<id>/` with a `meta.json` (label + timestamp). Every editor save creates one; restoring copies it back over the active files. |
| **Active config** | The config the app loads on launch, stored in `QSettings` as an identity (`ActiveConfig/name` + `ActiveConfig/source`), re-resolved to a directory at startup so a stale absolute path (e.g. from a deleted git worktree) can't point nowhere. Switching it warns, saves the session, and restarts. |
| **Startup fallback chain** | `build_station_with_fallback()` tries the active config, then its shipped baseline, then the always-loadable `sim_cryostat`, so a corrupted config surfaces a warning instead of crashing startup. Each candidate is checked by `validate_config_dir()` (a dry-run that imports classes and resolves driver refs without instantiating hardware). |
| **Orchestrator** | L3. The state machine that executes a procedure: it ticks, ramps system VIs toward targets, waits, triggers measurements, and enforces safety. Talks only to the Station. Monitoring runs in its tick loop, gated by `start_monitoring()`/`stop_monitoring()` — see **Monitoring (Orchestrator)**. |
| **Procedure** | L4. A declarative measurement sequence (e.g. field sweep, temperature sweep) subclassing `BaseProcedure`. Declares *what* to do via the four-method interface; the Orchestrator decides *how*. Procedures never import drivers or VIs. Most measurement procedures are **generic sweep procedures** (see below), so adding a measurement usually means adding a measurement VI, not a procedure. |
| **Generic sweep procedure** | L4. `core.procedure.SweepMeasureProcedure` — the "sweep one axis, run any measurement method" base, and the two axis procedures on it: `FieldSweep` ("Field Sweep") and `TemperatureSweep` ("Temperature Sweep"). ONE procedure per sweep axis runs ANY station **measurement method**, chosen in the GUI via a **structural** `measurement_vi` parameter whose `choices` come from `Station.measurement_vi_names()`. The base owns measurement-VI selection, station-driven measurement-parameter merging, `DataSchema` assembly, per-datapoint validation, and the four-method loop; a concrete axis procedure supplies only the ramp targets, the axis read-back, and the settle times (its `_initial_system_targets` / `_step_targets` / `_standby_targets` / `_axis_readback` / `_initiate_wait_s` / `_step_wait_s` hooks). Replaces the former one-procedure-per-measurement classes (`FieldSweepDC`, `FieldSweepIV`, `TemperatureSweepDC`). |
| **Data Manager** | L5. Writes measurement data to HDF5 files. Standalone: knows nothing about instruments. Created by procedures in `initiate()`. |
| **Session Manager (L6)** | `cryosoft/session/` — the experiment-management layer between core and the GUI. `SessionManager` (the façade, `session/manager.py`) owns the **experiment** lifecycle, records every **run** automatically from the Orchestrator's **run manifests**, installs the experiment's **session envelope** on the Orchestrator, and supplies `experiment_context()` for stamping HDF5 files. Single writer of experiment state, exactly as the Orchestrator is the single writer to hardware. Imported by gui/main only; imports core downward (contracts C11/C12). |
| **GUI** | PyQt6 windows (monitor, procedure, instrument panels). Talks to the system only through the Orchestrator and Station APIs; never imports drivers or concrete VIs. |

## Runtime concepts

| Term | Definition |
|---|---|
| **Tick / tick loop** | The Orchestrator's periodic heartbeat (`tick_interval_ms` in `monitor.yaml`). Each tick processes queued GUI actions and advances the state machine/ramps; while **Monitoring (Orchestrator)** is active it also polls VI state and checks safety. |
| **Monitoring (Orchestrator)** | The hardware-touching part of the tick: polling every VI into `states_updated`, the safety watchdog, stale detection, and the operational-status record. OFF at construction so a fresh launch polls nothing until the instruments are initiated; started/stopped via `Orchestrator.start_monitoring()`/`stop_monitoring()` (the Monitor window's header toggle). `run_procedure()` auto-starts it; stopping is refused outside IDLE/ERROR. `shutdown()` stops the tick timer entirely (app exit / test teardown). |
| **System targets** | The `targets` field of a `PhasePlan`/`StepPlan`: a `{"vi_name": Target(...)}` mapping telling the Station which system VIs to ramp where. (Pre-Wave-2 this was a nested dict `{"vi_name": {"target": ..., "rate": ...}}`.) |
| **Measurement commands** | The `commands` field of a `PhasePlan`: an ordered `tuple[Command, ...]` the Station dispatches to measurement VIs. (Pre-Wave-2 this was a dict `{"vi_name": {"method": kwargs}}`.) |
| **Plan** | Umbrella term for the typed, frozen-dataclass currency in `core/plan.py` that procedures, the Orchestrator, and the Station exchange, replacing the old untyped dicts/tuples. Concretely: `Target`, `Command`, `PhasePlan`, `StepPlan` (plus the GUI/data-facing `ParamSpec`, `ParamGroup`, `DataSchema`). Each validates eagerly at construction, so a malformed plan fails at the boundary that built it. |
| **Target** | `core/plan.py`. A frozen dataclass naming a desired end-state for one system variable: `target` (SI value), optional `rate` (ramp rate forwarded to the VI's `start_ramp()` only when not `None`) and optional `persistent` (magnet persistent-mode flag). |
| **Command** | `core/plan.py`. A frozen dataclass naming one method call to dispatch on a VI: `vi_name`, `method`, `kwargs` (defensively copied). The typed request a procedure hands the Orchestrator, dispatched via `Station.send_measurement_commands()`. |
| **PhasePlan** | `core/plan.py`. What `initiate()` and `standby()` return: `targets` (name→`Target`), `commands` (ordered `tuple[Command, ...]`, never reordered), and `wait_s` (settle seconds). |
| **StepPlan** | `core/plan.py`. What `change_sweep_step()` returns for the next sweep point (`targets` + `wait_s`), or `None` when the sweep is done. |
| **ParamSpec** | `core/plan.py`. One GUI-facing procedure-parameter declaration (`type`, `default`, `unit`, bounds or enumerated `choices`, `structural`, `widget_hint`). Semantic only — names no Qt widget class. Procedures declare their `sweep_parameters` / `system_parameters` / `measurement_parameters` as `ParamSpec`s; the GUI maps each to a Qt widget in `gui/param_form.py`. |
| **Structural parameter** | A `ParamSpec` with `structural=True`: changing its value changes *which* parameter groups the form has, so the GUI must re-derive the form. The generic sweep procedures' `measurement_vi` is the only structural parameter today — switching the measurement VI swaps the measurement parameter group. `ProcedureWindow` re-renders only the changed group boxes (diffing `get_param_groups` output by `ParamGroup.key`), leaving the sweep/system boxes and the axis widget as the same widget instances. |
| **ParamGroup** | `core/plan.py`. One rendered sub-panel of a procedure's parameter form: a stable `key` (for value caching across re-renders), a `title`, and a name→`ParamSpec` `params` map. Returned by `BaseProcedure.get_param_groups(station, selections)` and rendered one `QGroupBox` per group by the GUI. A generic sweep procedure appends a `measurement_select` group (the `measurement_vi` selector) and a `measurement:{vi_name}` group (the selected VI's parameters) after its static Sweep/System groups. |
| **DataSchema** | `core/plan.py`. The declared HDF5 layout of one run (`sweep_columns` dtypes + `arrays` lengths); offers `multiplexed(routes, scalar_columns=())` (per-route array expansion, plus per-route expansion of the named scalar columns) and `validate(datapoint)` (reports every mismatch at once). A generic sweep procedure assembles one in `initiate()` (axis column + system columns + the selected VI's arrays and scalar columns) and validates every datapoint against it in `BaseProcedure._save_datapoint`; a mismatch raises `DataSchemaError`, which the Orchestrator contains to ERROR before anything is written. |
| **Ramp / RampableVI** | Gradual, rate-limited approach to a setpoint (field, temperature). VIs that support it implement the `RampableVI` interface (`start_ramp`, `advance_ramp`, `ramp_status`). |
| **`@monitored` / `@control`** | Decorators from `core/decorators.py` marking VI methods. `@monitored`: read-only, polled every tick into `get_state()`. `@control`: user-callable action, rendered as a button/input in the GUI instrument panel. |
| **Control-validation standard** | The binding contract (full text in `BaseVirtualInstrument`'s docstring) that every user-facing `@control` action is validated before hardware is touched and always yields an explicit verdict. (1) Numeric bounds: the VI declares `control_limits = {method: {param: limit_name}}` and its `__init__` populates `self._limits[limit_name] = (lo, hi)` from the config's `init_params` — limits are setup properties (e.g. `max_temperature_K`, `max_ramp_rate_K_per_min`, `max_current`), so a new cryostat changes YAML, not code. Enforcement is inherited: `__init_subclass__` wraps every `@control` method; an out-of-range call raises `CryoSoftSafetyError` with the reason. Subclasses adding limits must merge (`{**Parent.control_limits, ...}`). (2) Semantic rules that aren't ranges (e.g. never energise the switch heater across a PSU/coil current mismatch) are explicit checks at the top of the method, raising `CryoSoftSafetyError`. (3) Verdict: the Orchestrator emits `action_succeeded(vi, method)` or `action_failed(vi, method, reason)` for every submitted action; `MonitorWindow` shows failures in the notification banner and successes as a transient status-bar message. Machine-enforced by the `test_conformance.py` control-validation tests: declared limits must name real `@control` methods/params, must be populated by every sim-buildable config, and must actually reject an out-of-range call. |
| **vi_type (class)** | Class attribute of a VI describing its device role: `magnet`, `temperature`, `measurement`, `level`. Set by the typed base classes. |
| **vi_type (config/registry)** | The *registry* role a VI plays in a Station, set in `devices.yaml`: `system` (rampable, receives system targets), `measurement` (receives measurement commands), `level` (safety monitoring), or `switch` (a **Switch VI** that multiplexes measurement channels by route). Distinct from the class `vi_type`. |
| **Persistent mode** | Superconducting-magnet operating mode where the field is held by a closed superconducting loop and the switch heater is off. Handled by `SuperconductingMagnetPersistentVI`. `start_ramp(target, persistent=True)`: `persistent=True` (default) ends the ramp by cooling the switch and parking the PSU at zero with the coil holding the field — for a single set-and-hold. `persistent=False` leaves the switch heater energised and the PSU holding the target directly — for a sequence of many ramps (e.g. a field sweep) that should only pay the heater warmup once. `get_field()`/`magnet_current()` read the coil current instead of the PSU current whenever the driver reports persistent mode. |
| **VISA / GPIB** | The instrument-communication standard (PyVISA library) and the bus most lab instruments use. A "resource string" like `GPIB0::19::INSTR` addresses one instrument. |
| **SI units rule** | All APIs use Tesla, Kelvin, Ampere, Volt, second. Display formatting (mK, µA) happens only in the GUI. |
| **Sweep builder** | `core/sweep_builder.py` — reusable sweep-array construction shared by Procedures: `build_piecewise_sweep()` (a `SweepSegment(start, end, step)` list stitched into one array, letting different sub-ranges use different step sizes, e.g. coarse outside a region and fine inside it), `load_custom_sweep_csv()` (arbitrary non-uniform sweep from a single-column CSV), and `apply_hysteresis()` (extends a one-directional sweep into a forward+backward loop). See **Sweep axis** for the declarative way most Procedures use these three. |
| **Experiment** | A named group of **runs** on one sample toward one scientific question, owned by the Session Manager (L6): `ExperimentRecord` holds the **user**, the sample-info snapshot, the config identity, the **session envelope**, the run list, free-text findings, and (publishing track) the ELN linkage. Persisted as `<data_dir>/experiments/<experiment_id>/experiment.json` so the record archives with the data; the `active.json` pointer resumes the open experiment across restarts. One experiment ↔ one ELN entry. |
| **Run** | One execution of one procedure, producing exactly one HDF5 file. Typed as `RunRecord` (L6): id, procedure, kind (`run`/`probe`), params, data-file path, timestamps, terminal status (`done`/`failed`/`aborted`), and the initiate-time instrument-settings snapshot. Recorded automatically from the Orchestrator's **run manifests**; runs outside an open experiment are not recorded (their HDF5 file still exists, unstamped). |
| **User** | The person measuring — identity, not authentication. `User` records (id, name, email, ORCID, ELN identity) live in the setup-local roster `users.json` next to the app settings, managed by `UserRoster`. Experiments reference a roster `user_id`, and the id/name are stamped into every run's HDF5 `experiment_info`. |
| **Attendance** | The per-experiment attended/unattended flag (`ExperimentRecord.attended`, set via `SessionManager.set_attended()`). Governs how much autonomy debug agents get in the agent-native track: gentle recovery actions are reserved for unattended sessions; attended sessions get diagnose-and-report only. |
| **Run manifest** | The dict the Orchestrator emits on `run_started` (once per run, after a successful `initiate()`) and re-emits on `run_finished` (with `finished_utc`, terminal `status` done/aborted/failed, and `reason` added). Carries `run_id`, `procedure`, `kind` (`run`; probe runs will carry `probe`), `params`, `data_file` (the HDF5 path, captured at start because the procedure closes its file before the run ends), and `started_utc`. The session layer records runs from these signals; a run whose setup fails emits no manifest. |
| **Session envelope** | `core/plan.py`'s `SessionEnvelope` (+ `EnvelopeBound`): per-experiment safety bounds narrower than the config limits, keyed by system VI name. Config `init_params` limits protect the *instrument* (setup property, YAML); the envelope protects the *sample* mounted for one experiment (experiment property, set by the session layer via `Orchestrator.set_session_envelope()`). Enforced in the Orchestrator so it binds every writer — targets are validated before dispatch (violation ⇒ the plan is rejected), and each bound with a `state_key` is checked against the VI's live reading every tick (violation ⇒ EMERGENCY, exactly like a tripped safety flag). |
| **Sweep axis** | `core/sweep_builder.py`'s `SweepAxis` dataclass + `sweep_axis_param_specs()` + `build_axis_sweep()` — the systematic, low-code way a Procedure gets sweep-shape flexibility (linear range / piecewise segments for a fine subfield / custom CSV, each optionally with hysteresis). A Procedure declares one `sweep_axis = SweepAxis(key=..., unit=..., data_key=..., description=..., default_start=..., default_end=..., default_steps=...)` class attribute (see `FieldSweep`, `TemperatureSweep`); `BaseProcedure.__init_subclass__` merges its seven hidden parameters (`{key}_mode`, `{key}_start`, `{key}_end`, `{key}_steps`, `{key}_segments`, `{key}_csv_path`, `{key}_hysteresis`) into `cls.parameters`, and the default `_build_sweep_array()` delegates to `build_axis_sweep()` — no per-procedure sweep-array code is needed. `gui/sweep_axis_widget.py`'s `SweepAxisWidget` renders the matching mode-selector form (Linear / Segments / CSV + a hysteresis checkbox) automatically in `ProcedureWindow`; a new sweep procedure never needs its own GUI code, only the `sweep_axis` declaration. |

## GUI concepts

| Term | Definition |
|---|---|
| **Quadrant layout** | The fixed 2x2 grid both `MonitorWindow` and `ProcedureWindow` use below their header/banner: a `main_splitter` (`QSplitter(Horizontal)`) containing `left_splitter` and `right_splitter` (each `QSplitter(Vertical)`), giving four panes whose boundaries are draggable but which can never be closed, detached, or floated — unlike the `QDockWidget` layout it replaced, which let panels overlap, hide each other, and become glitchy to resize. In `MonitorWindow`: top-left Instruments (scrollable 2-column `InstrumentPanel` grid), top-right Trends, bottom-left Sample Info, bottom-right Other Devices/Log (behind a `QComboBox` selector). In `ProcedureWindow`: top-left the parameter form (filling the quadrant, not capped at a fixed height), top-right the Queue, bottom-left/right Plot 1/Plot 2. Splitter proportions persist via `QSettings` alongside window geometry, restored defensively — see `MonitorWindow._restore_splitter_state()`'s shape-mismatch guard, which discards a restored blob whose orientation/child-count doesn't match (protects against a stale value under a reused settings key, e.g. from a since-removed earlier layout). |
| **Form autosave** | `cryosoft/gui/form_autosave.py` — the GUI's persisted form content (sample metadata, data dir, last procedure + params, run queue), written to one JSON file so a reopened app restores what the physicist typed. Historically named the "session model" (`gui/session.py`); renamed so **session** unambiguously means the L6 Session Management concept. The classes (`SessionState`) and the `last_session.json` file keep their old names for compatibility. |
| **MonitorHistory** | `cryosoft/gui/monitor_history.py`. A Qt-free, pure-Python ring buffer that accumulates flattened `{vi_name}_{field_name}` time-series history from `Orchestrator.states_updated`, one bounded `deque` per key, for the Trends panels' plots. |
| **Trends grid** | 1–4 `TrendPlotPanel`s arranged in the Trends quadrant's `QGridLayout`, auto-recomputed to `ceil(sqrt(N))` columns on every add/remove (1→1x1, 2→2x1, 3–4→2x2). The quadrant's "Add trend plot" button adds panels up to a cap of 4 (disabling once reached); each panel's own remove button drops it, never below a floor of 1. The two default panels opportunistically select a temperature/level reading (not a setting/rate field) once `MonitorHistory` has data. Each panel's Y-variable `QComboBox` uses `SizeAdjustPolicy.AdjustToContents` and a layout stretch factor of 1 so long flat keys (e.g. `temperature_sample_temperature`) get first claim on the row's width, ahead of the time-window selector and remove button. |
| **LifecycleToggleButton** | `cryosoft/gui/lifecycle_toggle.py`. A single state-dependent button + status glow dot replacing the old separate Initiate/Standby `QPushButton` pair, shared by `InstrumentPanel` (in its custom header row next to the VI name, since `QGroupBox` can't embed a widget next to its native title) and `MonitorWindow`'s Other Devices rows. Clicking only submits the opposite action (`"initiate"`/`"standby"`) via a callback — it never flips state optimistically. `set_initiated(bool)` is the only thing that changes the displayed state (text, icon, dot colour), and both call sites wire it to `Orchestrator.action_succeeded`, so the glow reflects confirmed instrument state, not a hopeful click. |

## Development harness

| Term | Definition |
|---|---|
| **Layer contracts (C1–C12)** | The machine-checked import rules in `pyproject.toml` `[tool.importlinter]`. Run with `make contracts`. A broken contract means the change crosses a layer boundary — route through the proper interface instead of editing the contract. C9/C10 isolate the troubleshoot toolbox in both directions; C11/C12 place the session layer (L6) between core and the GUI. |
| **Troubleshoot toolbox** | `cryosoft/troubleshoot/` — the diagnostic package (engine + CLI) for commissioning and debugging a setup. A leaf entry point beside the stack, like `main.py`: it may import drivers and the Station's config helpers; nothing in cryosoft imports it. Run while the main app is closed (serial ports are exclusive-open). CLI: `python -m cryosoft.troubleshoot scan/probe/check/methods/idn/read/write/query/send`, each one-shot with `--json` and exit code 0/1; `write`/`query`/`send` are the permission-gated verbs. Every invocation appends to the transcript `cryosoft/logs/troubleshoot.jsonl`. |
| **Fault taxonomy / FaultCode** | The stable machine-readable outcome codes every troubleshoot probe returns (`OK`, `CONFIG_INVALID`, `ADDRESS_NOT_ON_BUS`, `OPEN_FAILED`, `NO_RESPONSE`, `WRONG_IDN`, `GARBLED_RESPONSE`, `DRIVER_ERROR`). Codes are API for the triage skill: never renamed, only extended. |
| **Config preflight** | `check_config()` in the troubleshoot engine: for every `real_drivers` entry, verify bus presence, construct the driver, call `get_idn()`, and optionally match the config's `expect_idn` substring. The "is everything connected?" check run before a cooldown. |
| **Driver bench** | `DriverBench` in the troubleshoot engine: wraps one driver instance for method introspection, read/write-gated calls, and raw SCPI query/send. The primitive behind driver development and per-instrument debugging. |
| **`expect_idn`** | Optional key on a `real_drivers` entry in `devices.yaml`: a case-insensitive substring the instrument's `get_idn()` reply must contain. Lets the preflight detect swapped addresses/cables (`WRONG_IDN`). |
| **`get_idn()` (driver contract)** | Every driver — sim and real — exposes `get_idn()` taking no arguments, returning an identification string. Pre-SCPI Oxford instruments implement it with their native `V` command. Enforced by conformance. |
| **setup.md** | Per-config ground-truth document (`<config dir>/setup.md`): instrument purposes, wiring notes, dated known quirks, per-setup safe-testing limits. Written once during commissioning (human-confirmed), maintained by agents after every diagnosis. Template in `.claude/skills/setup-commission/references/`. |
| **Safe-testing ladder** | The excitation discipline for diagnostics (`.claude/skills/setup-supervisor/references/safe-testing.md`): L0 passive → L1 zero-excitation set/read-back → L2 minimal excitation (≤10 mA magnet lead current, ≤1 nA precision sources) → L3 operational values (human-confirmed state only). Instruments are not code: damage is irreversible. |
| **Incident report** | `cryosoft/logs/incidents/YYYY-MM-DD-<slug>.md` — the structured "cannot conclude" exit of the setup-supervisor skill (evidence table, layers ruled out, hypothesis with confidence, recommended next steps). A good report is a successful outcome; a guess is not. |
| **Conformance tests** | `tests/test_conformance.py`. Auto-discover every driver, VI, procedure, and config and check they follow their interface contracts. A new module is covered automatically, with no test-writing needed. |
| **`make check`** | The blocking quality gate (lint + contracts + tests). Run it before declaring any work done; CI runs exactly the same targets. |
| **`hardware` marker** | Pytest marker for tests needing physical instruments. Excluded by `make test` and CI; run manually at the cryostat. |
| **LOGBOOK.md** | The running development log at the project root (newest first, not git-tracked). Every work session — human or agent — ends by prepending an entry. |
